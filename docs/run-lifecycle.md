# Run Lifecycle

## Purpose

This document describes what a run does today, based on the code in:

- `src/lib/commands.mjs`
- `src/lib/run-state.mjs`

It reflects current behavior, not future intent.

## Run creation

`runProject()` creates a run from a validated project spec and factory config.

Artifacts written into the run directory include:

- `spec.snapshot.json`
- `execution-plan.json`
- `execution-plan.md`
- `roles.json`
- `run-state.json`
- `report.md`
- `task-briefs/*.md`

## Run status values

The run can be in one of these states:

- `planned`
- `in_progress`
- `completed`
- `attention_required`

Current status inference works like this:

- `attention_required` if any task is `failed`
- `completed` if every task is `completed`
- `in_progress` if any task is `in_progress`
- otherwise `planned`

## Task status values

Each task can be:

- `pending`
- `ready`
- `waiting_retry`
- `in_progress`
- `completed`
- `failed`
- `blocked`

## Initial task layout

When a run is created, the task ledger contains:

- one planning task: `planning-brief`
- one implementation task per core feature: `implement-*`
- one review task per core feature: `review-*`
- one verification task per core feature: `verify-*`
- one delivery task: `delivery-package`

Initial status rules:

- `planning-brief` starts as `ready`
- all other tasks start as `pending`

## Dependency flow

Dependencies are strict and unlock only when upstream tasks are marked `completed`.

Current dependency pattern:

- every `implement-*` depends on `planning-brief`
- every `review-*` depends on its matching `implement-*`
- every `verify-*` depends on its matching `review-*`
- `delivery-package` depends on all `verify-*` tasks

## How tasks become ready

`refreshRunState()` scans the task ledger and upgrades a task from `pending` to `ready` when all of its dependencies are already `completed`.

It does not automatically:

- mark work as completed
- inspect git diff quality
- infer task outcomes from arbitrary files outside the dispatch result contract

It only unlocks downstream tasks based on ledger state.

## Task updates

`updateTaskInRunState()` updates one task at a time.

Supported next statuses are:

- `pending`
- `ready`
- `waiting_retry`
- `in_progress`
- `completed`
- `failed`
- `blocked`

Current behavior:

- `attempts` increases only when the task is moved to `failed`
- an optional note is appended with a timestamp
- the run is refreshed after the change
- `report.md` is regenerated by the command layer

Hybrid/manual retry behavior:

- `retry` moves a task into `waiting_retry`
- `waiting_retry` records `retryCount`, `nextRetryAt`, and `lastRetryReason`
- once `nextRetryAt` has passed, the next state refresh promotes the task back to `ready`
- if the configured retry limit is reached, the task is escalated to `blocked`

## Reporting

`reportProjectRun()` refreshes the run state and rewrites `report.md`.

The report includes:

- run summary counts
- role configuration
- task ledger
- next actions
- risk stop rules
- definition of done

## Handoffs and dispatch in the lifecycle

`handoff` creates artifacts only for tasks already marked `ready`.

Current behavior:

- `handoff` creates artifacts only for tasks already marked `ready`
- `dispatch` runs or skips generated launchers
- `dispatch` validates any written result artifact against the expected JSON contract
- `dispatch` writes dispatch result files
- `dispatch execute` writes back to `run-state.json` when the parent run directory is available
- `dispatch execute` regenerates `report.md` when `execution-plan.json` is present
- `result` lets hybrid or manual tasks use the same artifact contract and state-sync flow
- `retry` lets transient hybrid-surface failures wait and automatically return to `ready` later

The loop is only partially closed automatically.

A human or higher-level automation still needs to decide when to move tasks through:

- `in_progress`
- retries
- re-planning
- manual or hybrid runtime follow-up

## Practical sequence

Today, a normal run looks like this:

1. Create the run.
2. Complete `planning-brief`.
3. Refresh or report the run so implementation tasks are visible as ready.
4. Generate handoffs for ready tasks.
5. Dispatch those handoffs.
6. Let `dispatch execute` sync `completed`, `failed`, or `blocked` outcomes when result artifacts are present.
7. Repeat for review, verification, and delivery.

## Example

```bash
node src/index.mjs run examples/project-spec.valid.json runs demo-run
node src/index.mjs task runs/demo-run/run-state.json planning-brief completed
node src/index.mjs handoff runs/demo-run/run-state.json
node src/index.mjs dispatch runs/demo-run/handoffs/index.json dry-run
node src/index.mjs report runs/demo-run/run-state.json
```
